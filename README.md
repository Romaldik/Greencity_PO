# GreenCity UI Automation Project
---
## Мета проекту
Закріпити навички автоматизації UI-тестування шляхом реалізації тест-кейсів за допомогою **Python + Selenium**. Основний фокус зроблено на переході від лінійних скриптів до професійної архітектури **Page Object Model (POM)**, що забезпечує гнучкість, читабельність та легку підтримку коду.
---
## Технологічний стек
* **Мова:** Python 3.12.0
* **Бібліотека:** Selenium WebDriver
* **Браузер:** Google Chrome 
* **Архітектура:** Page Object Model (POM) + Component-based approach
* **Тест-фреймворк:** pytest
* **Звітність:** Allure report 
* **Середовище:** venv (Virtual Environment)
---
## Методологія та архітектура
У проекті реалізовано чіткий поділ на логічні рівні:
1. **Base Component & Base Page:** Базові класи, що містять спільні методи (кліки, робота з модальними вікнами, пошук елементів).
2. **Page Objects:** Класи сторінок (`EventsPage`, `BasePage`), які описують локатори та методи взаємодії з конкретними сторінками сайту.
3. **Components:** Окремі класи для складних або повторюваних елементів, таких як картки подій (`EventCardComponent`) або модальні вікна (`SignUpModal`).
---
### Стратегія вибору селекторів:
* **Відносні XPath:** Використовуються всередині компонентів (наприклад, `./app-events-list-item...`) для точного пошуку елементів у межах конкретного об'єкта.
* **Contains & Normalize-space:** Для стабільної роботи з елементами, текст або класи яких можуть містити зайві пробіли або динамічні префікси Angular (`ng-`).
* **CSS Selectors:** Для швидкого доступу до статичних елементів навігації та хедера.

## Тест-кейси

### **TC-01: Реєстрація нового користувача**
* **Логіка:** Тестування форми створення акаунта з перевіркою валідації та обробкою системних помилок.
* **Реалізація:** Використання `try-except` блоку разом з `WebDriverWait` для гнучкої перевірки появи алертів про помилки.

### **TC-02: Фільтрація подій за тегами**
* **Логіка:** Взаємодія з динамічними списками Angular Material.
* **Реалізація:** Перевірка відображення контейнера `role='listbox'` перед вибором опції (`mat-option`).

### **TC-03: Пошук події за назвою**
* **Логіка:** Емуляція пошукового запиту через глобальний пошук та верифікація результатів.
* **Особливість:** Використано символ переносу рядка `\n` у методі `send_keys`, що імітує натискання клавіші `Enter` без необхідності пошуку окремої кнопки "Знайти".

##  Обробка стабільності (Waits)
Для забезпечення стабільності на динамічному Angular-сайті було застосовано комбінований підхід:
* **Implicit Waits:** Встановлено базове очікування на 10 секунд.
* **Explicit Waits (WebDriverWait):** Використовуються для перевірки видимості модальних вікон та станів кнопок перед взаємодією.
* **Відмова від sleep():** Код оптимізовано для роботи без жорстких пауз, що пришвидшує виконання тестів.
---

## Звітність (Allure Report)
У проекті інтегровано Allure Framework для деталізації результатів тестування:
* **Анотації:** Використано `@allure.feature`, `@allure.story` та `@allure.step` для структуризації звіту.
* **Аналіз помилок:** У разі падіння тесту Allure дозволяє відстежити конкретний крок, на якому стався збій, та переглянути логи WebDriver.
* **Генерація:** Звіт генерується динамічно з JSON-файлів, що збираються в папку `allure-results`.

##  Запуск проекту

1. **Активація віртуального середовища:**
   ```powershell 
   .\venv\Scripts\activate.ps1
   ```
2. **Встановлення залежностей:**
    ```powershell 
    pip install -r requirements.txt
    ```
3. **Запуск Тестів**
    ```powershell 
    pytest --alluredir=allure-results
    ```
4. **Отримання результатів Allure**
    ```powershell
    allure serve allure-results
    ```
